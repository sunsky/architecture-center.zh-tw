# <a name="index-table-pattern"></a><span data-ttu-id="305d6-104">索引資料表模式</span><span class="sxs-lookup"><span data-stu-id="305d6-104">Index Table pattern</span></span>

[!INCLUDE [header](../_includes/header.md)]

<span data-ttu-id="305d6-105">針對資料存放區中查詢經常參考的欄位建立索引。</span><span class="sxs-lookup"><span data-stu-id="305d6-105">Create indexes over the fields in data stores that are frequently referenced by queries.</span></span> <span data-ttu-id="305d6-106">此模式可以改進查詢效能，方法是讓應用程式更快找到要從資料存放區擷取的資料。</span><span class="sxs-lookup"><span data-stu-id="305d6-106">This pattern can improve query performance by allowing applications to more quickly locate the data to retrieve from a data store.</span></span>

## <a name="context-and-problem"></a><span data-ttu-id="305d6-107">內容和問題</span><span class="sxs-lookup"><span data-stu-id="305d6-107">Context and problem</span></span>

<span data-ttu-id="305d6-108">許多資料存放區會使用主索引鍵來組織實體集合的資料。</span><span class="sxs-lookup"><span data-stu-id="305d6-108">Many data stores organize the data for a collection of entities using the primary key.</span></span> <span data-ttu-id="305d6-109">應用程式可以使用此索引鍵以找出並擷取資料。</span><span class="sxs-lookup"><span data-stu-id="305d6-109">An application can use this key to locate and retrieve data.</span></span> <span data-ttu-id="305d6-110">此圖顯示保存客戶資訊的資料存放區範例。</span><span class="sxs-lookup"><span data-stu-id="305d6-110">The figure shows an example of a data store holding customer information.</span></span> <span data-ttu-id="305d6-111">主索引鍵是客戶識別碼。</span><span class="sxs-lookup"><span data-stu-id="305d6-111">The primary key is the Customer ID.</span></span> <span data-ttu-id="305d6-112">此圖顯示依主索引鍵 (客戶識別碼) 組織的客戶資訊。</span><span class="sxs-lookup"><span data-stu-id="305d6-112">The figure shows customer information organized by the primary key (Customer ID).</span></span>

![圖 1 - 依主索引鍵 (客戶識別碼) 組織的客戶資訊](./_images/index-table-figure-1.png)

<span data-ttu-id="305d6-114">主索引鍵對於根據此索引鍵的值擷取資料之查詢相當重要，如果應用程式必須根據某些其他欄位來擷取資料，則應用程式可能無法使用主索引鍵。</span><span class="sxs-lookup"><span data-stu-id="305d6-114">While the primary key is valuable for queries that fetch data based on the value of this key, an application might not be able to use the primary key if it needs to retrieve data based on some other field.</span></span> <span data-ttu-id="305d6-115">在客戶範例中，如果應用程式單純藉由參考某些其他屬性的值 (例如客戶所在的城鎮) 來查詢資料，則應用程式無法使用客戶識別碼主索引鍵來擷取客戶。</span><span class="sxs-lookup"><span data-stu-id="305d6-115">In the customers example, an application can't use the Customer ID primary key to retrieve customers if it queries data solely by referencing the value of some other attribute, such as the town in which the customer is located.</span></span> <span data-ttu-id="305d6-116">若要執行這類的查詢，應用程式必須擷取並檢查每個客戶記錄，這樣可能會讓處理程序變慢。</span><span class="sxs-lookup"><span data-stu-id="305d6-116">To perform a query such as this, the application might have to fetch and examine every customer record, which could be a slow process.</span></span>

<span data-ttu-id="305d6-117">許多關聯式資料庫管理系統支援次要索引。</span><span class="sxs-lookup"><span data-stu-id="305d6-117">Many relational database management systems support secondary indexes.</span></span> <span data-ttu-id="305d6-118">次要索引是個別的資料結構，由一或多個非主要 (次要) 索引鍵欄位所組織，它會指出每個索引值的資料儲存之位置。</span><span class="sxs-lookup"><span data-stu-id="305d6-118">A secondary index is a separate data structure that's organized by one or more nonprimary (secondary) key fields, and it indicates where the data for each indexed value is stored.</span></span> <span data-ttu-id="305d6-119">次要索引中的項目通常會依照次要索引鍵的值來儲存，以便快速查閱資料。</span><span class="sxs-lookup"><span data-stu-id="305d6-119">The items in a secondary index are typically sorted by the value of the secondary keys to enable fast lookup of data.</span></span> <span data-ttu-id="305d6-120">這些索引通常會由資料庫管理系統自動維護。</span><span class="sxs-lookup"><span data-stu-id="305d6-120">These indexes are usually maintained automatically by the database management system.</span></span>

<span data-ttu-id="305d6-121">您可以建立您需要之數目的次要索引，以支援您的應用程式執行的不同查詢。</span><span class="sxs-lookup"><span data-stu-id="305d6-121">You can create as many secondary indexes as you need to support the different queries that your application performs.</span></span> <span data-ttu-id="305d6-122">例如，在客戶識別碼是主索引鍵之關聯式資料庫的 [客戶] 資料表中，如果應用程式經常依據客戶所在的城鎮查閱客戶，則在城鎮欄位新增次要索引很有幫助。</span><span class="sxs-lookup"><span data-stu-id="305d6-122">For example, in a Customers table in a relational database where the Customer ID is the primary key, it's beneficial to add a secondary index over the town field if the application frequently looks up customers by the town where they reside.</span></span>

<span data-ttu-id="305d6-123">不過，雖然次要索引在關聯式系統中很常見，但是雲端應用程式所使用的大部分 NoSQL 資料存放區不提供對等的功能。</span><span class="sxs-lookup"><span data-stu-id="305d6-123">However, although secondary indexes are common in relational systems, most NoSQL data stores used by cloud applications don't provide an equivalent feature.</span></span>

## <a name="solution"></a><span data-ttu-id="305d6-124">解決方法</span><span class="sxs-lookup"><span data-stu-id="305d6-124">Solution</span></span>

<span data-ttu-id="305d6-125">如果資料存放區不支援次要索引，您可以藉由建立您自己的索引資料表，以手動模擬它們。</span><span class="sxs-lookup"><span data-stu-id="305d6-125">If the data store doesn't support secondary indexes, you can emulate them manually by creating your own index tables.</span></span> <span data-ttu-id="305d6-126">索引資料表會依據指定的索引鍵來組織資料。</span><span class="sxs-lookup"><span data-stu-id="305d6-126">An index table organizes the data by a specified key.</span></span> <span data-ttu-id="305d6-127">三個策略經常用於建構索引資料表，取決於所需的次要索引數目，以及應用程式執行之查詢的本質。</span><span class="sxs-lookup"><span data-stu-id="305d6-127">Three strategies are commonly used for structuring an index table, depending on the number of secondary indexes that are required and the nature of the queries that an application performs.</span></span>

<span data-ttu-id="305d6-128">第一個策略是複製每個索引資料表中的資料，但是依據不同索引鍵來組織 (完整反正規化)。</span><span class="sxs-lookup"><span data-stu-id="305d6-128">The first strategy is to duplicate the data in each index table but organize it by different keys (complete denormalization).</span></span> <span data-ttu-id="305d6-129">下圖顯示索引資料表，依據 Town 和 LastName 來組織相同的客戶資訊。</span><span class="sxs-lookup"><span data-stu-id="305d6-129">The next figure shows index tables that organize the same customer information by Town and LastName.</span></span>

![圖 2 - 資料在每個索引資料表中複製](./_images/index-table-figure-2.png)

<span data-ttu-id="305d6-131">如果資料相較於使用每個索引鍵查詢的次數相對靜態，則此策略是適當的。</span><span class="sxs-lookup"><span data-stu-id="305d6-131">This strategy is appropriate if the data is relatively static compared to the number of times it's queried using each key.</span></span> <span data-ttu-id="305d6-132">如果資料更加動態，維護每個索引資料表的處理額外負荷會變得太大而無法使用這種方法。</span><span class="sxs-lookup"><span data-stu-id="305d6-132">If the data is more dynamic, the processing overhead of maintaining each index table becomes too large for this approach to be useful.</span></span> <span data-ttu-id="305d6-133">此外，如果資料量非常大，儲存重複資料所需的空間量也會非常大。</span><span class="sxs-lookup"><span data-stu-id="305d6-133">Also, if the volume of data is very large, the amount of space required to store the duplicate data is significant.</span></span>

<span data-ttu-id="305d6-134">第二個策略是建立依據不同索引鍵組織的正規化索引資料表，並且藉由使用主索引鍵來參考原始資料，而不是複製它，如下圖所示。</span><span class="sxs-lookup"><span data-stu-id="305d6-134">The second strategy is to create normalized index tables organized by different keys and reference the original data by using the primary key rather than duplicating it, as shown in the following figure.</span></span> <span data-ttu-id="305d6-135">原始資料稱為事實資料表。</span><span class="sxs-lookup"><span data-stu-id="305d6-135">The original data is called a fact table.</span></span>

![圖 3 - 資料是由每個索引資料表參考](./_images/index-table-figure-3.png)

<span data-ttu-id="305d6-137">這個技術會節省空間，並減少維護重複資料的額外負荷。</span><span class="sxs-lookup"><span data-stu-id="305d6-137">This technique saves space and reduces the overhead of maintaining duplicate data.</span></span> <span data-ttu-id="305d6-138">缺點是應用程式必須執行兩個查閱作業以使用次要索引鍵尋找資料。</span><span class="sxs-lookup"><span data-stu-id="305d6-138">The disadvantage is that an application has to perform two lookup operations to find data using a secondary key.</span></span> <span data-ttu-id="305d6-139">它必須在索引資料表中尋找資料的主索引鍵，然後使用主索引鍵查閱事實資料表中的資料。</span><span class="sxs-lookup"><span data-stu-id="305d6-139">It has to find the primary key for the data in the index table, and then use the primary key to look up the data in the fact table.</span></span>

<span data-ttu-id="305d6-140">第三個策略是建立依據不同索引鍵 (經常複製擷取的欄位) 組織的部分正規化索引資料表。</span><span class="sxs-lookup"><span data-stu-id="305d6-140">The third strategy is to create partially normalized index tables organized by different keys that duplicate frequently retrieved fields.</span></span> <span data-ttu-id="305d6-141">參考事實資料表以存取不常存取的欄位。</span><span class="sxs-lookup"><span data-stu-id="305d6-141">Reference the fact table to access less frequently accessed fields.</span></span> <span data-ttu-id="305d6-142">下圖顯示經常存取資料如何在每個索引資料表中複製。</span><span class="sxs-lookup"><span data-stu-id="305d6-142">The next figure shows how commonly accessed data is duplicated in each index table.</span></span>

![圖 4 - 在每個索引資料表中複製經常存取資料](./_images/index-table-figure-4.png)

<span data-ttu-id="305d6-144">您可以使用這個策略，打破前兩個策略之間的平衡。</span><span class="sxs-lookup"><span data-stu-id="305d6-144">With this strategy, you can strike a balance between the first two approaches.</span></span> <span data-ttu-id="305d6-145">空間與維護額外負荷不像複製整個資料集那麼大，經常查詢的資料可以藉由使用單一查閱來快速擷取。</span><span class="sxs-lookup"><span data-stu-id="305d6-145">The data for common queries can be retrieved quickly by using a single lookup, while the space and maintenance overhead isn't as significant as duplicating the entire data set.</span></span>

<span data-ttu-id="305d6-146">如果應用程式經常藉由指定值的組合來查詢資料 (例如，「尋找住在 Redmond 且姓氏為 Smith 的所有客戶」)，您可以實作索引資料表中項目的索引鍵，作為 Town 屬性和 LastName 屬性的串連。</span><span class="sxs-lookup"><span data-stu-id="305d6-146">If an application frequently queries data by specifying a combination of values (for example, “Find all customers that live in Redmond and that have a last name of Smith”), you could implement the keys to the items in the index table as a concatenation of the Town attribute and the LastName attribute.</span></span> <span data-ttu-id="305d6-147">下圖顯示根據複合索引鍵的索引資料表。</span><span class="sxs-lookup"><span data-stu-id="305d6-147">The next figure shows an index table based on composite keys.</span></span> <span data-ttu-id="305d6-148">索引鍵會依據 Town 排序，然後針對具有相同 Town 值的記錄依據 LastName 排序。</span><span class="sxs-lookup"><span data-stu-id="305d6-148">The keys are sorted by Town, and then by LastName for records that have the same value for Town.</span></span>

![圖 5 - 根據複合索引鍵的索引資料表](./_images/index-table-figure-5.png)

<span data-ttu-id="305d6-150">索引資料表可以透過分區化資料加速查詢作業，在分區索引鍵是雜湊時特別有用。</span><span class="sxs-lookup"><span data-stu-id="305d6-150">Index tables can speed up query operations over sharded data, and are especially useful where the shard key is hashed.</span></span> <span data-ttu-id="305d6-151">下圖顯示範例，其中分區索引鍵是客戶識別碼的雜湊。</span><span class="sxs-lookup"><span data-stu-id="305d6-151">The next figure shows an example where the shard key is a hash of the Customer ID.</span></span> <span data-ttu-id="305d6-152">索引資料表可以依據非雜湊值 (Town 和 LastName) 來組織資料，並提供雜湊分區索引鍵作為查閱資料。</span><span class="sxs-lookup"><span data-stu-id="305d6-152">The index table can organize data by the nonhashed value (Town and LastName), and provide the hashed shard key as the lookup data.</span></span> <span data-ttu-id="305d6-153">如果應用程式必須擷取落在範圍內的資料，或者必須以非雜湊索引鍵的順序擷取資料，這樣可以讓應用程式免於重複計算雜湊索引鍵 (昂貴的作業)。</span><span class="sxs-lookup"><span data-stu-id="305d6-153">This can save the application from repeatedly calculating hash keys (an expensive operation) if it needs to retrieve data that falls within a range, or it needs to fetch data in order of the nonhashed key.</span></span> <span data-ttu-id="305d6-154">例如，「尋找住在 Redmond 的所有客戶」這類的查詢可以藉由在索引資料表中找出相符項目以快速解析，這些項目會儲存在連續區塊中。</span><span class="sxs-lookup"><span data-stu-id="305d6-154">For example, a query such as “Find all customers that live in Redmond” can be quickly resolved by locating the matching items in the index table, where they're all stored in a contiguous block.</span></span> <span data-ttu-id="305d6-155">然後，使用儲存在索引資料表中的分區索引鍵，遵循客戶資料的參考。</span><span class="sxs-lookup"><span data-stu-id="305d6-155">Then, follow the references to the customer data using the shard keys stored in the index table.</span></span>

## <a name="issues-and-considerations"></a><span data-ttu-id="305d6-157">問題和考量</span><span class="sxs-lookup"><span data-stu-id="305d6-157">Issues and considerations</span></span>

<span data-ttu-id="305d6-158">當您決定如何實作此模式時，請考慮下列幾點：</span><span class="sxs-lookup"><span data-stu-id="305d6-158">Consider the following points when deciding how to implement this pattern:</span></span>

- <span data-ttu-id="305d6-159">維護次要索引的額外負荷可能相當巨大。</span><span class="sxs-lookup"><span data-stu-id="305d6-159">The overhead of maintaining secondary indexes can be significant.</span></span> <span data-ttu-id="305d6-160">您必須分析及了解應用程式使用的查詢。</span><span class="sxs-lookup"><span data-stu-id="305d6-160">You must analyze and understand the queries that your application uses.</span></span> <span data-ttu-id="305d6-161">只有在索引資料表會經常使用時才建立索引資料表。</span><span class="sxs-lookup"><span data-stu-id="305d6-161">Only create index tables when they're likely to be used regularly.</span></span> <span data-ttu-id="305d6-162">請勿建立理論式索引資料表來支援應用程式不會執行，或只是偶爾執行的查詢。</span><span class="sxs-lookup"><span data-stu-id="305d6-162">Don't create speculative index tables to support queries that an application doesn't perform, or performs only occasionally.</span></span>
- <span data-ttu-id="305d6-163">複製索引資料表中的資料，會增加維護資料多個複本所需之儲存體成本和工作的巨大額外負荷。</span><span class="sxs-lookup"><span data-stu-id="305d6-163">Duplicating data in an index table can add significant overhead in storage costs and the effort required to maintain multiple copies of data.</span></span>
- <span data-ttu-id="305d6-164">以參考原始資料的正規化結構實作索引資料表，需要應用程式執行兩個查閱作業來尋找資料。</span><span class="sxs-lookup"><span data-stu-id="305d6-164">Implementing an index table as a normalized structure that references the original data requires an application to perform two lookup operations to find data.</span></span> <span data-ttu-id="305d6-165">第一個作業會搜尋索引資料表以擷取主索引鍵，第二個作業會使用主索引鍵來擷取資料。</span><span class="sxs-lookup"><span data-stu-id="305d6-165">The first operation searches the index table to retrieve the primary key, and the second uses the primary key to fetch the data.</span></span>
- <span data-ttu-id="305d6-166">如果系統在非常大的資料集上併入一些索引資料表，會難以在索引資料表與原始資料之間維護一致性。</span><span class="sxs-lookup"><span data-stu-id="305d6-166">If a system incorporates a number of index tables over very large data sets, it can be difficult to maintain consistency between index tables and the original data.</span></span> <span data-ttu-id="305d6-167">也許可以憑藉最終一致性模型來設計應用程式。</span><span class="sxs-lookup"><span data-stu-id="305d6-167">It might be possible to design the application around the eventual consistency model.</span></span> <span data-ttu-id="305d6-168">例如，若要插入、更新或刪除資料，應用程式可以將訊息張貼至佇列，並讓個別工作執行作業及維護會以非同步方式參考這個資料的索引資料表。</span><span class="sxs-lookup"><span data-stu-id="305d6-168">For example, to insert, update, or delete data, an application could post a message to a queue and let a separate task perform the operation and maintain the index tables that reference this data asynchronously.</span></span> <span data-ttu-id="305d6-169">如需實作最終一致性的詳細資訊，請參閱[資料一致性入門](https://msdn.microsoft.com/library/dn589800.aspx)。</span><span class="sxs-lookup"><span data-stu-id="305d6-169">For more information about implementing eventual consistency, see the [Data Consistency Primer](https://msdn.microsoft.com/library/dn589800.aspx).</span></span>

   >  <span data-ttu-id="305d6-170">Microsoft Azure 儲存體資料表針對相同資料分割 (稱為實體群組交易) 中保存的資料之變更，支援交易式更新。</span><span class="sxs-lookup"><span data-stu-id="305d6-170">Microsoft Azure storage tables support transactional updates for changes made to data held in the same partition (referred to as entity group transactions).</span></span> <span data-ttu-id="305d6-171">如果您可以在相同資料分割中儲存事實資料表以及一或多個索引資料表的資料，可以使用這項功能來協助確保一致性。</span><span class="sxs-lookup"><span data-stu-id="305d6-171">If you can store the data for a fact table and one or more index tables in the same partition, you can use this feature to help ensure consistency.</span></span>

- <span data-ttu-id="305d6-172">索引資料表本身可能是資料分割或分區化。</span><span class="sxs-lookup"><span data-stu-id="305d6-172">Index tables might themselves be partitioned or sharded.</span></span>

## <a name="when-to-use-this-pattern"></a><span data-ttu-id="305d6-173">使用此模式的時機</span><span class="sxs-lookup"><span data-stu-id="305d6-173">When to use this pattern</span></span>

<span data-ttu-id="305d6-174">使用此模式以在應用程式經常需要使用主索引鍵 (或分區索引鍵) 以外的索引鍵擷取資料時，改善查詢效能。</span><span class="sxs-lookup"><span data-stu-id="305d6-174">Use this pattern to improve query performance when an application frequently needs to retrieve data by using a key other than the primary (or shard) key.</span></span>

<span data-ttu-id="305d6-175">此模式可能不適合下列時機︰</span><span class="sxs-lookup"><span data-stu-id="305d6-175">This pattern might not be useful when:</span></span>

- <span data-ttu-id="305d6-179">選取作為索引資料表之次要索引鍵的欄位資料值平衡有高度偏差。</span><span class="sxs-lookup"><span data-stu-id="305d6-179">The balance of the data values for a field selected as the secondary key for an index table are highly skewed.</span></span> <span data-ttu-id="305d6-180">例如，如果 90% 的記錄在欄位中包含相同值，則建立及維護索引資料表以根據此欄位查閱資料，會建立比循序掃描資料更多的額外負荷。</span><span class="sxs-lookup"><span data-stu-id="305d6-180">For example, if 90% of the records contain the same value in a field, then creating and maintaining an index table to look up data based on this field might create more overhead than scanning sequentially through the data.</span></span> <span data-ttu-id="305d6-181">不過，如果非常頻繁地查詢位於剩餘 10% 記錄中的目標值，則索引很有用。</span><span class="sxs-lookup"><span data-stu-id="305d6-181">However, if queries very frequently target values that lie in the remaining 10%, this index can be useful.</span></span> <span data-ttu-id="305d6-182">您應該了解您的應用程式執行的查詢，以及查詢的執行頻率。</span><span class="sxs-lookup"><span data-stu-id="305d6-182">You should understand the queries that your application is performing, and how frequently they're performed.</span></span>
