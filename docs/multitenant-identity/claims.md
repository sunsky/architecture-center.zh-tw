## <a name="issuer-validation"></a><span data-ttu-id="765d5-183">簽發者驗證</span><span class="sxs-lookup"><span data-stu-id="765d5-183">Issuer validation</span></span>

<span data-ttu-id="765d5-184">在 OpenID Connect 中，簽發者宣告 ("iss") 會識別發行識別碼權杖的 IDP。</span><span class="sxs-lookup"><span data-stu-id="765d5-184">In OpenID Connect, the issuer claim ("iss") identifies the IDP that issued the ID token.</span></span> <span data-ttu-id="765d5-185">OIDC 驗證流程的其中一部份，是驗證簽發者宣告是否符合實際的簽發者。</span><span class="sxs-lookup"><span data-stu-id="765d5-185">Part of the OIDC authentication flow is to verify that the issuer claim matches the actual issuer.</span></span> <span data-ttu-id="765d5-186">OIDC 中介軟體會為您處理這部分。</span><span class="sxs-lookup"><span data-stu-id="765d5-186">The OIDC middleware handles this for you.</span></span>

<span data-ttu-id="765d5-187">在 Azure AD 中，每個 AD 租用戶的簽發者值是唯一的 (`https://sts.windows.net/<tenantID>`)。</span><span class="sxs-lookup"><span data-stu-id="765d5-187">In Azure AD, the issuer value is unique per AD tenant (`https://sts.windows.net/<tenantID>`).</span></span> <span data-ttu-id="765d5-188">因此，應用程式應該執行額外的檢查，確保簽發者代表的租用戶允許登入應用程式。</span><span class="sxs-lookup"><span data-stu-id="765d5-188">Therefore, an application should do an additional check, to make sure the issuer represents a tenant that is allowed to sign in to the app.</span></span>

<span data-ttu-id="765d5-189">針對單一租用戶應用程式，您可以只檢查簽發者是否為您擁有的租用戶。</span><span class="sxs-lookup"><span data-stu-id="765d5-189">For a single-tenant application, you can just check that the issuer is your own tenant.</span></span> <span data-ttu-id="765d5-190">實際上，OIDC 中介軟體預設會自動執行此動作。</span><span class="sxs-lookup"><span data-stu-id="765d5-190">In fact, the OIDC middleware does this automatically by default.</span></span> <span data-ttu-id="765d5-191">在多租用戶應用程式中，您需要允許多個簽發者，對應到不同的租用戶。</span><span class="sxs-lookup"><span data-stu-id="765d5-191">In a multi-tenant app, you need to allow for multiple issuers, corresponding to the different tenants.</span></span> <span data-ttu-id="765d5-192">以下是常用的方法：</span><span class="sxs-lookup"><span data-stu-id="765d5-192">Here is a general approach to use:</span></span>

* <span data-ttu-id="765d5-193">在 OIDC 中介軟體選項中，將 **ValidateIssuer** 設為 false。</span><span class="sxs-lookup"><span data-stu-id="765d5-193">In the OIDC middleware options, set **ValidateIssuer** to false.</span></span> <span data-ttu-id="765d5-194">這會關閉自動檢查。</span><span class="sxs-lookup"><span data-stu-id="765d5-194">This turns off the automatic check.</span></span>
* <span data-ttu-id="765d5-195">當租用戶註冊時，會將租用戶及簽發者儲存到您的使用者 DB 中。</span><span class="sxs-lookup"><span data-stu-id="765d5-195">When a tenant signs up, store the tenant and the issuer in your user DB.</span></span>
* <span data-ttu-id="765d5-196">每當使用者登入時，便會查閱資料庫中的簽發者。</span><span class="sxs-lookup"><span data-stu-id="765d5-196">Whenever a user signs in, look up the issuer in the database.</span></span> <span data-ttu-id="765d5-197">如果找不到簽發者，則表示該租用戶尚未註冊。</span><span class="sxs-lookup"><span data-stu-id="765d5-197">If the issuer isn't found, it means that tenant hasn't signed up.</span></span> <span data-ttu-id="765d5-198">您可以將他們重新導向至註冊頁面。</span><span class="sxs-lookup"><span data-stu-id="765d5-198">You can redirect them to a sign up page.</span></span>
* <span data-ttu-id="765d5-199">您也可以將特定租用戶設為封鎖清單；例如，沒有為他們的訂用帳戶付費的客戶。</span><span class="sxs-lookup"><span data-stu-id="765d5-199">You could also blacklist certain tenants; for example, for customers that didn't pay their subscription.</span></span>

<span data-ttu-id="765d5-200">如需更詳細的討論，請參閱[多租用戶應用程式中的租用戶註冊和登入][signup]。</span><span class="sxs-lookup"><span data-stu-id="765d5-200">For a more detailed discussion, see [Sign-up and tenant onboarding in a multitenant application][signup].</span></span>

## <a name="using-claims-for-authorization"></a><span data-ttu-id="765d5-201">使用宣告進行授權</span><span class="sxs-lookup"><span data-stu-id="765d5-201">Using claims for authorization</span></span>

<span data-ttu-id="765d5-202">利用宣告，使用者的身分識別不再是整合型實體。</span><span class="sxs-lookup"><span data-stu-id="765d5-202">With claims, a user's identity is no longer a monolithic entity.</span></span> <span data-ttu-id="765d5-203">例如，使用者可能有電子郵件地址、電話號碼、生日、性別等等。可能使用者的 IDP 會儲存所有這類資訊。</span><span class="sxs-lookup"><span data-stu-id="765d5-203">For example, a user might have an email address, phone number, birthday, gender, etc. Maybe the user's IDP stores all of this information.</span></span> <span data-ttu-id="765d5-204">但當您驗證使用者時，您通常會取得這些資訊的子集做為宣告。</span><span class="sxs-lookup"><span data-stu-id="765d5-204">But when you authenticate the user, you'll typically get a subset of these as claims.</span></span> <span data-ttu-id="765d5-205">在此模型中，使用者的身分識別只是宣告的組合。</span><span class="sxs-lookup"><span data-stu-id="765d5-205">In this model, the user's identity is simply a bundle of claims.</span></span> <span data-ttu-id="765d5-206">當您進行有關使用者的授權決策時，您會尋找特定的宣告集。</span><span class="sxs-lookup"><span data-stu-id="765d5-206">When you make authorization decisions about a user, you will look for particular sets of claims.</span></span> <span data-ttu-id="765d5-207">換句話說，問題「使用者 X 可以執行動作 Y 嗎」最後會變成「使用者 X 是否有宣告 Z」。</span><span class="sxs-lookup"><span data-stu-id="765d5-207">In other words, the question "Can user X perform action Y" ultimately becomes "Does user X have claim Z".</span></span>

<span data-ttu-id="765d5-208">以下是一些檢查宣告的基本模式。</span><span class="sxs-lookup"><span data-stu-id="765d5-208">Here are some basic patterns for checking claims.</span></span>

* <span data-ttu-id="765d5-209">若要檢查使用者是否有含有特定值的特定宣告：</span><span class="sxs-lookup"><span data-stu-id="765d5-209">To check that the user has a particular claim with a particular value:</span></span>

   ```csharp
   if (User.HasClaim(ClaimTypes.Role, "Admin")) { ... }
   ```

   <span data-ttu-id="765d5-210">此程式碼會檢查使用者是否有含有「Admin」的角色宣告。</span><span class="sxs-lookup"><span data-stu-id="765d5-210">This code checks whether the user has a Role claim with the value "Admin".</span></span> <span data-ttu-id="765d5-211">它會正確地處理使用者沒有角色宣告，或有多個角色宣告的情況。</span><span class="sxs-lookup"><span data-stu-id="765d5-211">It correctly handles the case where the user has no Role claim or multiple Role claims.</span></span>
  
   <span data-ttu-id="765d5-212">**ClaimTypes** 類別定義常用宣告類型的常數。</span><span class="sxs-lookup"><span data-stu-id="765d5-212">The **ClaimTypes** class defines constants for commonly-used claim types.</span></span> <span data-ttu-id="765d5-213">不過，您可以使用任何字串值做為宣告類型。</span><span class="sxs-lookup"><span data-stu-id="765d5-213">However, you can use any string value for the claim type.</span></span>
* <span data-ttu-id="765d5-214">當您預期最多只有一個值時，若要取得宣告類型的單一值：</span><span class="sxs-lookup"><span data-stu-id="765d5-214">To get a single value for a claim type, when you expect there to be at most one value:</span></span>

  ```csharp
  string email = User.FindFirst(ClaimTypes.Email)?.Value;
  ```

* <span data-ttu-id="765d5-215">取得宣告類型的所有值：</span><span class="sxs-lookup"><span data-stu-id="765d5-215">To get all the values for a claim type:</span></span>

  ```csharp
  IEnumerable<Claim> groups = User.FindAll("groups");
  ```
